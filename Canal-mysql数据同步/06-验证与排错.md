### 6.1.3 整表行数对账

如果两边库都有数据（之前已经同步过），先做行数对账：

```bash
# 在 RDS 上执行 (用 systemdba 账号)
docker run --rm mysql:8.0.36 \
  mysql -h rm-bp1c2x3191qxo5o7x8o.mysql.rds.aliyuncs.com \
        -u systemdba -p'gRdrqp3LLuSgdgZH' jielong \
        -e "SELECT TABLE_NAME, TABLE_ROWS FROM information_schema.TABLES WHERE TABLE_SCHEMA='jielong' ORDER BY TABLE_NAME;"

# 在本地 MySQL 上执行
docker exec sig-mysql-txg mysql -uroot -p'Aw123456.' jielong \
  -e "SELECT TABLE_NAME, TABLE_ROWS FROM information_schema.TABLES WHERE TABLE_SCHEMA='jielong' ORDER BY TABLE_NAME;"
```

> `TABLE_ROWS` 在 InnoDB 下是估算值，**不会完全一致**（误差 10% 内正常）。
> 精确对账需要逐表 `COUNT(*)`，慢但准。

### 6.1.4 增量同步测试（标准流程）

```sql
-- Step 1: RDS 上 - 建一张测试表
USE jielong;
CREATE TABLE _canal_smoke_test (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  payload VARCHAR(200),
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- Step 2: RDS 上 - 插 3 条
INSERT INTO _canal_smoke_test(payload) VALUES('a'),('b'),('c');
COMMIT;

-- Step 3: RDS 上 - 更新 1 条
UPDATE _canal_smoke_test SET payload='A' WHERE id=1;
COMMIT;

-- Step 4: RDS 上 - 删 1 条
DELETE FROM _canal_smoke_test WHERE id=2;
COMMIT;
```

10 秒内在本地验证：

```bash
docker exec sig-mysql-txg mysql -uroot -p'Aw123456.' jielong \
  -e "SELECT * FROM _canal_smoke_test ORDER BY id;"
```

期望：

```
+----+---------+---------------------+
| id | payload | created_at          |
+----+---------+---------------------+
|  1 | A       | 2026-06-30 ...      |   # UPDATE 已同步
|  3 | c       | 2026-06-30 ...      |   # DELETE 已同步 (id=2 已删)
+----+---------+---------------------+
```

如果 DDL+CRUD 全部对齐 → 整条链路通。清理：

```sql
-- RDS 上
DROP TABLE _canal_smoke_test;
-- 本地会自动跟着 DROP
```

## 6.3 常见问题分类排查

### 6.3.1 启动期 NPE（canal-server）

```
ERROR ... CanalStarter - start canal failed
java.lang.NullPointerException at com.alibaba.otter.canal.deployer.CanalStarter.start(CanalStarter.java:68)
```

**原因**：`canal.properties` 缺必填项。
**解决**：确认有这三行：

```properties
canal.serverMode = tcp
canal.destinations = jielong
canal.instance.global.spring.xml = classpath:spring/file-instance.xml
```

```
ERROR ... NullPointerException at CanalController.initInstanceConfig(CanalController.java:396)
```

**原因**：`canal.destinations` 与实例目录不匹配。
**解决**：

- `canal.properties` 里 `canal.destinations = jielong`
- 容器内路径必须是 `/home/admin/canal-server/conf/jielong/instance.properties`（不是 `jielong-instance.properties`）

### 6.3.2 RDS 连接问题

```
ERROR ... Communications link failure
ERROR ... Access denied for user 'systemdba'@'x.x.x.x'
```

**排查**：

```bash
# 1. canal-server 容器能否解析 RDS 域名
docker exec canal-server nslookup rm-bp1c2x3191qxo5o7x8o.mysql.rds.aliyuncs.com

# 2. 网络可达
docker exec canal-server nc -zv rm-bp1c2x3191qxo5o7x8o.mysql.rds.aliyuncs.com 3306

# 3. 账号能登
docker exec canal-server mysql -h rm-bp1c2x3191qxo5o7x8o.mysql.rds.aliyuncs.com \
  -u systemdba -p'gRdrqp3LLuSgdgZH' -e "SELECT 1;"

# 4. 是否有 binlog 权限
docker exec canal-server mysql -h rm-bp1c2x3191qxo5o7x8o.mysql.rds.aliyuncs.com \
  -u systemdba -p'gRdrqp3LLuSgdgZH' \
  -e "SHOW GRANTS; SHOW MASTER STATUS;"
```

**常见原因**：

- 阿里云 RDS 白名单没加 canal-server 出口 IP
- 账号没 `REPLICATION SLAVE` 权限
- canal-server 容器走 NAT，出口 IP 不是宿主机主 IP

### 6.3.3 同步延迟 / 积压

**症状**：RDS 上 INSERT 后本地分钟级才出现。

**排查**：

```bash
# 1. canal-server 是否在解析
tail -f /home/canal/canal-server-logs/jielong/jielong.log | grep -E 'received|filtered|sent'

# 2. adapter 是否在消费
docker compose logs --tail=200 canal-adapter | grep -E 'batch|sink|insert|update'

# 3. MySQL 是否慢
docker exec sig-mysql-txg mysql -uroot -p'Aw123456.' -e \
  "SELECT * FROM mysql.slow_log ORDER BY start_time DESC LIMIT 10\G"
```

**性能调优杠杆**（按收益排序）：

| 杠杆 | 当前值 | 调到 | 收益 | 风险 |
|---|---|---|---|---|
| `canal.tcp.batch.size` | 2000 | 5000 | adapter 单次拉更多消息 | 内存占用 |
| `syncBatchSize` | 5000 | 10000 | 一次提交批量大 | 内存占用 |
| `threads` (adapter) | 20 | 30-40 | 写入并发 | MySQL 锁争用 |
| `innodb_buffer_pool_size` | 28G | 35G | 更多热数据常驻 | 宿主机内存 |
| `rewriteBatchedStatements` | 已开 | — | 已最大化 | — |

### 6.3.4 ringbuffer / 并行解析报错

```
ERROR ... IllegalArgumentException: ringbuffer size must be a power of 2: 500
```

**原因**：`canal.instance.parser.parallelBufferSize` 不是 2 的幂。
**解决**：改为 256 / 512 / 1024 / 2048 之一。当前已设 512。

```
# parallelThreadSize 不要显式设
canal.instance.parser.parallel = true       # ✓ 显式开
canal.instance.parser.parallelBufferSize = 512   # ✓ 显式设
# canal.instance.parser.parallelThreadSize = 19   # ✗ 删掉, 让 fallback 自动算
```

### 6.3.5 表结构不同步（mirrorDb 没生效）

**症状**：RDS 上 ALTER 了某张表，本地没改。

**排查**：

```bash
# 1. 确认 jielong.yml 是 mirrorDb: true
cat /home/canal/canal-adapter-conf/rdb/jielong.yml | grep mirrorDb

# 2. 重启 adapter 让它重新读配置
docker compose restart canal-adapter

# 3. 看 adapter 启动日志是否报 DDL 错
docker compose logs --tail=300 canal-adapter | grep -i 'ddl\|alter\|create'
```

**常见原因**：

- 配置修改后没 reload
- RDS 表用了 MySQL 8 不支持的特性（很少见）
- 目标库已存在同名表但结构不同（adapter 不会自动 ALTER 已存在的表）

### 6.3.6 OOM / 频繁 Full GC

**症状**：adapter/server 容器重启 / 卡顿。

```bash
# 看 JVM 内存
docker exec canal-adapter jstat -gcutil 1 1000 5
# 期望: OYGCT 不超过 5%, YGC 频率正常
```

**解决**：

```bash
# 增大堆（编辑 docker-compose）
environment:
  JAVA_OPTS: "-server -Xms8g -Xmx8g -Xmn4g -XX:+UseG1GC ..."
  # 或 canal-server:
  CANAL_SERVER_HEAP: "-Xms8g -Xmx8g"

docker compose up -d canal-adapter canal-server
```

## 6.4 性能调优要点总结（本次已应用）

> 这些都是基于 canal 1.1.5 **源码** 验证的，详见 [alibaba/canal Wiki](https://github.com/alibaba/canal/wiki)。

| 组件 | 参数 | 默认值 | 本次值 | 出处 | 验证方式 |
|---|---|---|---|---|---|
| canal-server | `parallelBufferSize` | 256 | 512 | AbstractEventParser.java | 源码 `Assert.assertTrue((bufferSize & (bufferSize - 1)) == 0)` 要求 2 的幂 |
| canal-server | `parallelThreadSize` | fallback 60% 核数 | 不显式设 | file-instance.xml | Java fallback 自动算 |
| canal-server | `canal.auto.scan` | true | false | canal.properties | 关闭避免扫描镜像自带 example 实例 |
| canal-server | JVM 堆 | 2G | 4G | docker-compose env | 默认偏小 |
| canal-adapter | `canal.tcp.batch.size` | 500 | 2000 | application.yml | 单次拉取消息数 |
| canal-adapter | `syncBatchSize` | 1000 | 5000 | application.yml | 一次提交批量 |
| canal-adapter | `threads` | 5 | 20 | application.yml | 写入并发 |
| canal-adapter | `rewriteBatchedStatements` | 无 | true | jdbc.url | JDBC 合并 INSERT |
| canal-adapter | JVM 堆 | 1G | 4G | docker-compose env | 默认偏小 |
| 目标 MySQL | `innodb_buffer_pool_size` | 128M | 28G | docker-compose command | 吃满 70% 内存 |
| 目标 MySQL | `innodb_flush_log_at_trx_commit` | 1 | 2 | docker-compose command | 性能优先，崩溃丢 1 秒 |

## 6.6 长期监控建议

| 指标 | 阈值 | 报警动作 |
|---|---|---|
| canal-server 11111 端口不监听 | 连续 1 分钟 | 重启容器，通知 oncall |
| adapter `/syncSwitch/jielong` OFF | 立即 | 检查日志，开启同步 |
| RDS 到 canal-server 网络断 | 连续 3 分钟 | 检查白名单 / VPC |
| adapter 与 server 位点差 | 持续 > 10000 条 | 性能瓶颈，调参或加机器 |
| MySQL 慢查询数 / 分钟 | > 50 | 检查索引 / 写入瓶颈 |
| 任一容器 OOM 重启 | 立即 | 调大堆或加内存 |

## 6.7 应急回滚

如果同步出问题需要紧急回滚：

```bash
# 1. 停止同步链路（不停 MySQL）
docker compose -f /home/canal/docker-compose.yml stop canal-adapter canal-server

# 2. 用最近的全量备份恢复本地 jielong 库（如有）
# mysqldump / xtrabackup 还原

# 3. 修复后按顺序启动
docker compose -f /home/canal/docker-compose.yml start canal-server
sleep 30
docker compose -f /home/canal/docker-compose.yml start canal-adapter
```

> ⚠️ canal-server 重启后默认从**当前位点**开始拉（不是从上次位点），如果想从上次停止位置续传，需要保留 `meta.dat` 文件（在 `/home/admin/canal-server/conf/jielong/meta.dat`，对应宿主机挂载点）。
